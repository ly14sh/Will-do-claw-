# 2026-03-08 开发记录 (v8版本)

## 需求背景

本次开发围绕 Android 16 官方实况通知继续推进，但目标与 v7 的 `ProgressStyle` 实验不同：

1. **不再追求进度条样式**，回到“信息展示优先”
2. **保留小胶囊正常显示**，同时提升展开卡片的信息密度
3. **针对 OPPO / ColorOS 实机表现做反向验证**，按系统真实渲染规则调整字段映射
4. **重构胶囊文案分发逻辑**，让 Native / Flyme Provider 只负责渲染，不再各自维护一套业务判断

---

## 一、阶段目标

本轮的核心目标有两个：

- **先摸清 OPPO 到底吃哪些标准通知字段**
- **再把现有胶囊逻辑重构成“统一分发 + 平台渲染”架构**

为了避免盲改，开发时先加了一套实验室探针模式，通过真实胶囊在 OPPO 手机上的展示结果，反推出 OEM 对字段的采纳情况。

---

## 二、OPPO 实机验证结论

通过实验室探针与真实胶囊组合测试，最终确认 OPPO / ColorOS 当前这条原生实况通知路径有以下特点：

### 1. 小胶囊可以恢复，但构建方式非常敏感

一旦通知构建链路偏离原先可工作的模式，小胶囊就可能消失。最终确认更稳妥的方式是：

- 继续使用平台 `Notification.Builder`
- 保留 `setShortCriticalText(...)`
- 保留 `setRequestPromotedOngoing(...)`
- 保留原有 promoted extras 的兼容写法
- 同时保留 `contentText` 占位、`BigTextStyle` 兜底、`showWhen(true)` 等更接近旧实现的构造方式

### 2. OPPO 会稳定显示的字段有限

实测可稳定显示的主要字段为：

- `contentTitle`
- `contentText`
- `subText`
- `Action`

而以下字段价值较低或基本不稳定：

- `contentInfo`
- `BigTextStyle.bigText`
- `BigTextStyle.bigContentTitle`

这意味着 Android 16 原生卡片在 OPPO 上更接近**固定 2~3 行文本 + 按钮**，并不适合做复杂排版。

### 3. 探针残留文案会直接污染可见 UI

在探针模式下，`SUB 副标题`、`BTN 动作` 等占位文字会被 OPPO 直接展示出来，因此后续已移除这类对用户可见的探针占位内容。

---

## 三、最终产品策略

结合实机结果，本轮最终确定 Android 16 原生卡片采用：

### 1. 双行极简为主，必要时补第三行

- 第 1 行：核心识别信息
- 第 2 行：辅助识别信息
- 第 3 行：只在普通日程 / 课程需要补时间时使用

### 2. 不再显示这些噪音信息

- `进行中`
- `即将开始`
- 探针占位文案
- 无意义的空白说明文本

### 3. 普通日程时间补位规则

对于普通日程 / 课程：

- 若 `location` 和 `description` 都为空：第 2 行显示时间段 `HH:mm-HH:mm`
- 若存在 `location` 或 `description`：第 2 行显示详情，第 3 行显示时间

这样既能避免“只有一个标题显得太空”，又不会破坏取件 / 列车 / 打车这类强识别型胶囊的简洁性。

## 四、架构重构

本次最重要的代码层变化，不是单纯改字段，而是**重构胶囊消息分发架构**。

### 重构前的问题

原来逻辑分散在多个地方：

- `CapsuleStateManager` 负责挑选胶囊，也在拼接展示文案
- `NativeCapsuleProvider` 一边渲染，一边重新计算状态 / 按钮 / 跳转
- `FlymeCapsuleProvider` 也维护了另一套类似逻辑

结果是：

- Native / Flyme 行为容易漂移
- 字段映射调整成本很高
- 一处改动常常要同步改两个 Provider
- 很难针对 OPPO 做单独的展示策略收敛

### 重构后的结构

本轮新增统一展示模型与消息分发层：

#### 1. `CapsuleDisplayModel.kt`

新增路径：

- `app/src/main/java/com/antgskds/calendarassistant/service/capsule/CapsuleDisplayModel.kt`

职责：统一承载胶囊展示字段，例如：

- `shortText`
- `primaryText`
- `secondaryText`
- `tertiaryText`
- `expandedText`
- `action`
- `probeFields`

#### 2. `CapsuleMessageComposer.kt`

新增路径：

- `app/src/main/java/com/antgskds/calendarassistant/service/capsule/CapsuleMessageComposer.kt`

职责：根据事件类型统一产出展示模型。

它现在会复用：

- `TransportUtils` 处理列车 / 打车动态主文案
- `PickupUtils` 处理取件 / 取餐码拆解

并统一决定：

- 小胶囊显示什么
- 卡片第 1 行显示什么
- 卡片第 2 行显示什么
- 普通日程是否补第 3 行时间
- 是否显示动作按钮，以及按钮文案

#### 3. `CapsuleStateManager.kt`

职责收敛为：

- 决定哪些胶囊应该出现
- 负责聚合 / 过期 / notifId / 时间范围等状态计算
- 调用 `CapsuleMessageComposer` 生成展示数据

不再负责 Native / Flyme 平台层的具体显示细节。

#### 4. Provider 变成纯渲染层

- `NativeCapsuleProvider.kt`
- `FlymeCapsuleProvider.kt`

两者现在都直接消费统一的 `display model`，只负责：

- 把对应字段填进通知
- 处理平台专属渲染细节
- 保留厂商兼容 extras / RemoteViews / promoted ongoing 构建链路

## 五、字段映射结果

### 1. 取件 / 取餐

- 小胶囊：取件码 / 取餐码
- 第 1 行：取件码 / 取餐码
- 第 2 行：平台 + 地点（或地点）
- 按钮：`已取`

### 2. 打车

- 小胶囊：车牌号
- 第 1 行：车牌号
- 第 2 行：车型 `·` 颜色
- 按钮：`已用车`

### 3. 列车

- 小胶囊：检票口 / 座位号
- 第 1 行：检票口 / 座位号
- 第 2 行：`车次 -> 目的地`（若有目的地）
- 按钮：`已检票`

### 4. 普通日程 / 课程

- 小胶囊：标题
- 第 1 行：标题
- 第 2 行：地点或备注摘要；若两者都无则显示时间
- 第 3 行：仅在第 2 行已显示详情时补时间
- 按钮：`已完成`

## 九、后续建议

本轮重构已经完成主要目标，后续可以继续做的优化包括：

1. **继续微调普通日程第 2 / 第 3 行的展示优先级**
2. **视需要补充航班专属解析，而不是继续复用列车 fallback**

4. **当官方 Android 16 live update API 公开链路更稳定后，再决定是否彻底移除旧反射兼容路径**

## 开发花絮

- 这次最有价值的不是“改了一行显示什么”，而是把胶囊系统真正拆成了“状态计算”和“平台渲染”两层
- OPPO 实机测试再次说明：Android 官方通知 API 是否“支持”，和 OEM 最终如何“展示”，是两回事
- 最终方案不是追求更复杂的样式，而是接受系统约束后，把有限的 2~3 行信息利用到最好
