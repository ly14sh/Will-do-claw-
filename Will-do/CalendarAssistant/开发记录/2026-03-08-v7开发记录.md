# 2026-03-08 开发记录 (v7版本)

## 需求背景

本次开发围绕两个方向展开探索：

1. **ColorOS 流体云适配可行性评估**
2. **Android 16 官方实况通知 / ProgressStyle 模板实验**

最终结论：

- **ColorOS 流体云方案暂缓**：依赖 OPPO 企业申请能力与私有接入链路，当前不适合继续投入
- **Android 16 ProgressStyle 实验回滚**：虽然部分系统上可正常显示原生实况通知，但最终样式仍受 OEM SystemUI 强约束，未达到预期，因此本次代码未合入主分支

---

## 一、ColorOS 流体云方案评估

### 背景

原计划为 OPPO / ColorOS 适配流体云，并设计统一的数据契约，后续再对接模板与 UI 变量绑定。

### 调研结果

- 本地模板工程 `F:\Will-do` 可以确认是 OPlus/OPPO 卡片模板工程，说明前端模板侧并非不可做
- 但宿主 Android 侧真正发布流体云卡片，需要 Pantanal / Push / 模板 ID 等配套能力
- 经进一步确认，相关 SDK 与接入权限并非面向普通开发者直接开放，需要企业资质与申请流程

### 结论

本项目当前阶段**放弃 ColorOS 流体云接入**，后续等待官方开放更多公开接口后再评估。

## 二、Android 16 官方实况通知实验

### 实验目标

尝试用 Android 16 官方 `ProgressStyle` 替代当前部分原生实况通知反射调用，并验证以下思路：

- 使用官方模板承载通用胶囊展开态
- 进度条固定为 `0`，把它当作视觉分割线使用
- 底部继续保留操作按钮

### 实验分支

- 分支名：`feature/android16-progress-capsule`

### 主要实验文件

- `app/src/main/java/com/antgskds/calendarassistant/service/capsule/provider/NativeCapsuleProvider.kt`

---

## 三、实验过程与关键发现

### 1. 官方 API 具备可用性

项目已满足 Android 16 官方实况通知的基础条件：

- `compileSdk / targetSdk` 已为 36
- Manifest 已声明 `POST_PROMOTED_NOTIFICATIONS`
- Live Channel 重要性不是 `MIN`

因此从 API 角度看，项目具备尝试官方 promoted ongoing / live update 的前提。

### 2. NotificationCompat 路径会影响小胶囊显示

实验初期将原生通知改为 `NotificationCompat.Builder` 后，状态栏小胶囊消失。

后续调整为更接近原先可工作链路的方式：

- 回到平台 `Notification.Builder`
- 保留 `setShortCriticalText(...)`
- 保留 `setRequestPromotedOngoing(...)`
- 同时补充相关 extras

这样后，小胶囊在 OPPO 设备上重新出现。

### 3. Header / ContentText 信息容易重复

Android 16 官方模板中，Header 自身会显示较多系统信息，例如：

- 应用名
- 状态词（如“进行中”）
- 时间（如 `now`）

因此如果 `contentText` 继续显示时间或状态，会出现明显重复。

实验中最终将 `contentText` 调整为：

- 优先显示地点 / 细节
- 没有则回退为 `description`
- 再没有才显示“点击查看详情”

### 4. 官方按钮样式不可自定义

尝试将底部按钮做成“右侧胶囊按钮 + 背景块”的形式，但确认后发现：

- `ProgressStyle` 下按钮仍通过 `addAction(...)` 提供
- 对齐方式、背景、圆角、位置由 SystemUI 决定
- 若改成自定义布局，通常会失去 promoted/live 这条官方显示路径

因此**官方模板下无法稳定实现右侧胶囊形按钮**。

### 5. OPPO 上的进度条左侧会出现额外图标

这是本次实验最终放弃的重要原因。

在 OPPO / ColorOS 设备上：

- 原生实况通知可以显示
- 小胶囊也能恢复
- 但 `ProgressStyle` 的进度轨道左侧会出现一个类似“导航”的图标

经查证：

- 官方 API 确实提供 `setProgressStartIcon(...)`、`setProgressTrackerIcon(...)`、`setProgressEndIcon(...)`
- AOSP 标准行为下，这些图标位可以设为 `null`
- 但如果 OEM 仍然额外绘制左侧图标，则属于厂商 `SystemUI` 的实现差异，应用层无法稳定去除

换言之，`ProgressStyle` 虽是官方模板，但最终渲染并不是完全由应用控制。

## 四、最终结论

### 为什么本次实验最终回滚

虽然这次实验验证了不少重要结论：

- Android 16 官方实况通知 API 是可接入的
- 原生小胶囊可以通过更接近平台链路的方式恢复
- Header / Content / Action 的字段映射可以逐步调优

但仍存在一个无法接受的问题：

- **OEM 对 `ProgressStyle` 的渲染不可控**
- 在 OPPO 设备上，左侧额外图标破坏了预期的“分割线”视觉方案
- 该问题大概率不是业务代码层面可可靠解决的

因此本次实验决定：

- **不将 Android 16 ProgressStyle 方案合入主分支**
- **回滚全部实验代码**
- **删除实验分支**

## 六、本次探索的价值

虽然最终没有保留代码，但这次尝试非常有价值，至少明确了以下几点：

1. **ColorOS 流体云当前不适合本项目投入**
2. **Android 16 官方实况通知可以继续关注，但要接受 OEM 差异**
3. **官方模板适合做“系统认可的标准样式”，不适合强行做高度定制布局**
4. **如果后续继续做原生实况通知优化，应优先考虑信息映射，而不是视觉欺骗式布局**
