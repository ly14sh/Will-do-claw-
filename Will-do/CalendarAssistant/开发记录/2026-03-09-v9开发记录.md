# 2026-03-09 开发记录 (v9版本)

## 需求背景

本轮开发的主线不再是胶囊样式本身，而是围绕“系统日历反向同步”补齐一个此前长期缺失的能力：

1. **系统日历中的重复日程无法正确同步到 App**
2. **重复日程一旦直接平铺，会把全部列表、悬浮窗、主页和提醒链路全部冲乱**
3. **跨天事件在主页“明日安排”打开后会导致重复展示，甚至触发 Compose 重复 key 崩溃**
4. **重复日程一旦参与胶囊 / 通知 / 编辑，还必须明确“父节点”和“实际实例”到底谁参与行为**

因此，v9 的目标不是简单“支持重复日程”四个字，而是把**同步层、数据层、展示层、编辑层、提醒层、胶囊层**全部收束成一套能长期维护的方案。

---

## 一、本轮阶段目标

本轮的核心目标有四个：

- **让系统日历重复日程可以反向同步进 App**
- **避免重复实例刷爆全部列表和悬浮窗**
- **支持把重复系列中的某一次实例脱离出来单独编辑**
- **保证重复子实例可以正常参与胶囊 / 提醒，而重复父节点只负责聚合展示**

在这个目标下，产品上最终确定采用“**参考课表父子节点，但以系统日历 Instances 作为真实来源**”的方案，而不是在 App 内部重新实现一套 RRULE 解释器。

---

## 二、问题定位与最终判断

### 1. 原有反向同步为什么不支持重复日程

旧逻辑读取的是 `CalendarContract.Events`。

这对普通单次日程有效，但对重复日程只能拿到：

- 系列主记录
- RRULE 定义

拿不到“某一天真正发生的那一条 occurrence”。

结果就是：

- 每天 / 每周 / 每月 / 每年的系统重复日程无法正常进入 App
- 即使能读到系列主记录，也无法映射到现有 `MyEvent` 单次事件模型

### 2. 为什么不能直接把 180 天实例全部平铺

如果简单把重复系列展开成大量普通事件，会立刻产生这些问题：

- 全部列表页被重复实例刷屏
- 悬浮窗出现大量重复项
- 提醒和胶囊可能被过多实例淹没
- 编辑其中一条后，很难表达“只改这次还是改整个系列”

因此最终没有采用“纯实例平铺”的简化方案，而是引入父子节点语义。

---

## 三、最终产品决策

### 1. 重复日程同步模型

系统日历重复日程采用：

- **父节点**：重复系列摘要
- **子节点**：当前同步窗口内的实际实例

其中：

- 父节点只用于聚合展示
- 子节点才是真正参与今日列表、胶囊、提醒和时间判断的实体

### 2. 同步窗口

最终同步窗口从先前讨论的更大范围收缩为：

- **过去 30 天**
- **未来 30 天**

这样做的原因很明确：

- 降低一次反向同步的实例展开量
- 避免存储和 UI 压力
- 让胶囊 / 提醒链路保持可控

### 3. 全部列表页展示规则

- 普通事件：正常显示
- 重复父节点：每个系列只显示 1 条
- 重复子节点：不在全部列表中直接平铺

同时加入重复图标，强化“这是系列摘要，不是普通单次日程”的认知。

### 4. 编辑规则

当用户从“全部列表页”点击重复系列时：

- 默认基于**下次实例**进入编辑
- 保存后不会直接修改整个系统重复系列
- 而是生成一个**脱离系列的本地单次事件**
- 同时把原实例加入父节点排除列表，避免后续反向同步再次把这次实例导回来

这本质上就是把课表中的“影子课程 / 排除原实例”思路迁移到了重复日程上。

### 5. 设置策略

偏好设置最终增加了新开关：

- `同步重复日程 (Beta)`

并确定为：

- **默认关闭**
- **关闭时立即清理已导入的重复父/子节点**
- **开启时立即执行一次反向同步**

## 四、数据模型改造

### 1. `MyEvent` 扩展

为承载重复系列语义，`MyEvent` 新增了重复相关字段，包括：

- `isRecurring`
- `isRecurringParent`
- `recurringSeriesKey`
- `recurringInstanceKey`
- `parentRecurringId`
- `excludedRecurringInstances`
- `nextOccurrenceStartMillis`
- `skipCalendarSync`

语义划分如下：

- **重复父节点**：`isRecurring = true` 且 `isRecurringParent = true`
- **重复子节点**：`isRecurring = true` 且 `isRecurringParent = false`
- **脱离后的单次事件**：`isRecurring = false`

### 2. 工具层抽取

新增 `RecurringEventUtils.kt`，统一负责：

- 生成系列 key
- 生成父节点 ID
- 生成实例 key
- 生成实例 ID
- 计算事件起止毫秒值
- 格式化“下一次提醒时间”文案

这样避免重复系列相关 ID 规则散落在多个类里。

## 五、系统日历反向同步重构

### 1. 数据源切换到 `CalendarContract.Instances`

`CalendarManager` 新增基于 `CalendarContract.Instances` 的查询方法，专门读取时间窗口内的真实 occurrence。

同时保留普通 `Events` 查询，用于：

- 普通单次日程同步
- 查询重复系列定义

### 2. 重复系列和重复实例分两层处理

新的反向同步流程中：

- 普通日程仍走原有 `Events` 更新 / 删除 / 新增逻辑
- 重复日程单独进入“阶段三”处理

重复阶段会：

1. 查询重复系列定义
2. 查询窗口内重复实例
3. 按系列分组
4. 为每个系列生成 1 个父节点和若干子节点

### 3. 复合 ID 与实例唯一性

重复实例不再直接使用系统裸 `eventId`，而是使用：

- `seriesKey`
- `instanceKey = systemEventId + beginMillis`

本地实例 ID 则构造成：

- `recurring_instance_<instanceKey>`

这样可以避免每日重复、每周重复等所有 occurrence 共用同一个系统 ID 而互相覆盖。

### 4. 去重与防崩兜底

在本轮开发中，曾遇到 Compose 因重复 key 直接崩溃的问题。为此，系统额外增加了多层防重：

- 构建重复子节点时按 `instanceKey` 去重
- `AppRepository.updateEvents()` 存盘前按 `id` 去重
- 全部列表页展示前再按 `id` 兜底去重

这保证了即使系统日历返回重复实例，UI 也不会再因重复 key 崩溃。

## 六、下次实例时间的准确性修复

### 1. 早期问题

最初父节点的“下一次提醒时间”只是从当前 30 天窗口内的子节点推导。

这会导致：

- 每日重复看起来没问题
- 但每周 / 每两周 / 每月 / 每年 / 自定义间隔时，很容易拿到错误日期

例如用户设置“每周重复”，但窗口内实例与真实下一次 occurrence 的关系并不总是等价，最终可能显示成错误的日期。

### 2. 最终修复方式

最终改为：

- 先读取重复系列定义（`Events`）
- 再针对单个系列额外查询真正的“下一次 occurrence”

并根据 RRULE 的 `FREQ + INTERVAL` 估算搜索跨度：

- `DAILY`
- `WEEKLY`
- `MONTHLY`
- `YEARLY`
- 自定义 `INTERVAL`

这样就不再依赖 30 天子节点缓存来猜测下次时间，而是直接以系统日历展开结果为准。

最终效果是：

- 每天
- 每周
- 每两周
- 每月
- 每年
- 自定义 x 日 / 周 / 月 / 年

都能正确给出父节点上的“下一次提醒时间”。

## 七、UI 层调整

### 1. 全部列表页

`AllEventsPage` 调整为：

- 只展示重复父节点
- 用 `nextOccurrenceStartMillis` 显示“下次：yyyy-MM-dd HH:mm”
- 加入重复 icon

这样全部页从“实例平铺”变成“系列摘要列表”，可读性明显提升。

### 2. 编辑弹窗

`AddEventDialog` 在重复系列编辑时增加了底部说明区，展示：

- `此日程为重复日程`
- `下一次提醒时间：yyyy-MM-dd HH:mm`
- `本次修改将应用到下次实例，并脱离重复系列`

这让用户在进入编辑时能明确知道自己操作的不是整个系列，而是系列中的下一次实例。

### 3. 悬浮窗

悬浮窗最初仍然会平铺重复实例，后续已统一改成：

- 只保留普通事件和重复父节点
- 父节点按 `nextOccurrenceStartMillis` 排序
- 显示重复状态标签
- 重复父节点不提供完成 / 检票这类单次实例动作

### 4. 主页今日 / 明日预览

主页部分新增了两个关键修复：

#### (1) 跨天事件判定改为真正的时间区间重叠判断

不再简单使用 `date >= startDate && date <= endDate`，而是使用起止时间与当天区间重叠判断。

这修复了：

- `23:00 - 00:00` 错误同时出现在两天的问题

#### (2) “显示明日日程”导致跨天事件崩溃的修复

当跨天事件同时出现在今日和明日 section 时，两个 `LazyColumn items` 都使用相同 `event.id` 作为 key，会触发 Compose 重复 key 崩溃。

本轮最终采用“双修复”方案：

- 今日 section key 改为 `today_<id>`
- 明日 section key 改为 `tomorrow_<id>`
- 同时在 `MainViewModel` 中让“明日安排”过滤掉今天已经显示过的同一事件

最终产品策略是：

- **跨天日程只在今日安排显示，不在明日安排重复出现**

## 八、提醒、胶囊与正向同步策略

### 1. 父节点与子节点职责切分

本轮最重要的一个行为层决策是：

- **重复父节点永远只做展示，不参与提醒 / 胶囊 / 正向回写**
- **重复子节点才参与真实时间行为**

### 2. 胶囊问题修复

开发中曾一度为了防止通知洪泛，把所有重复日程都排除在胶囊外，导致重复事件虽然能显示在列表里，却不会在对应时间触发胶囊。

最终修正为：

- 只过滤父节点
- 保留重复子实例进入胶囊链路

因此现在：

- 重复系列能在全部页只显示 1 条摘要
- 但真正到时间时，仍由子实例触发胶囊

### 3. 普通提醒

普通提醒链路与胶囊调度共用 `NotificationScheduler`，本轮也同步收敛为：

- 父节点：不设提醒
- 子节点：可参与提醒
- 但前提是“同步重复日程”开关处于开启状态

### 4. App -> 系统日历回写

系统导入的重复父节点 / 子节点不参与 App -> 系统日历回写。

原因是当前系统写回链路只会写普通单次事件，不会写完整 RRULE；若强行回写，极易把重复系列“打平”为普通单次事件。

因此：

- 父节点：只读聚合
- 子节点：只读同步结果
- 脱离后的单次事件：才按普通本地事件处理

## 九、偏好设置改造

### 1. 新增重复同步开关

在“日历”设置板块下新增：

- `同步重复日程 (Beta)`

它只在主“日历同步”开启时显示。

### 2. 开启主同步时立即同步

原先主“日历同步”开启时主要做的是正向同步，现在改成更明确的链路：

1. 启用同步
2. 立即执行一次 `App -> 系统日历`
3. 立即执行一次 `系统日历 -> App`
4. 初始化日历内容观察者

这样第一次开启后，系统已有事件和 App 已有数据都能立刻统一，而不是等待后续定时轮询。

### 3. 开启重复同步时立即反向同步

当用户打开“同步重复日程 (Beta)”时：

- 立刻保存设置
- 立刻执行一次反向同步

因此不需要再等 1 分钟定时器。

### 4. 关闭重复同步时立即清理

当用户关闭“同步重复日程 (Beta)”时：

- 立刻删除所有已导入的重复父节点
- 立刻删除所有已导入的重复子节点
- 取消这些节点对应的提醒
- 刷新胶囊状态

但保留：

- 用户从重复系列中脱离出来的普通单次事件

这保证开关的语义对用户来说是直观的：

- **关了就立即消失**
- **不是等下次同步才生效**

### 5. observer 与 1 分钟定时器的关系

本轮没有移除现有同步机制，而是保留双保险：

- `CalendarContentObserver`：系统日历变化后约 2 秒防抖同步
- `AlarmManager` 每 1 分钟：兜底同步

因此设置页“立即同步”解决的是开关切换时的即时性，observer 解决的是平时系统日历修改后的准实时同步。

## 十、与课表逻辑的关系

本轮设计明确参考了课表现有机制，但没有直接复制：

- 课表主课程 -> 重复日程父节点
- 影子课程 -> 脱离系列的单次事件
- `excludedDates` -> `excludedRecurringInstances`

但二者也有本质差异：

- 课表是本地规则生成
- 重复日程是系统日历 `Instances` 展开的真实结果

因此 v9 并没有在 App 内部重写整套 RRULE 引擎，而是把系统日历当作唯一可信来源。

## 十四、当前限制与后续建议

虽然 v9 已完成核心目标，但仍有几个值得后续继续打磨的点：

1. **继续观察系统日历 observer 的稳定性**

   - 当前已与胶囊链路对齐，但仍建议实机继续验证边界行为

3. **如未来要支持真正的“编辑整个重复系列”**
   - 则需要补 RRULE 级写回，而不是继续沿用“脱离为单次事件”策略

4. **可以考虑为重复系列补更强的可视化区分**
   - 例如更明确的副文案、颜色或系列摘要布局

## 开发花絮

- 这次最核心的价值不是“让重复日程能显示出来”，而是把“系列”和“实例”彻底分开，避免后面所有功能都语义混乱
- 课表逻辑这次提供了很好的结构参考，但最终仍然必须以系统日历 `Instances` 为真实来源，不能偷懒自己猜 RRULE
- 真正麻烦的从来不是“查出来”，而是“查出来之后，列表怎么展示、胶囊怎么参与、编辑怎么脱离、关闭开关怎么清理”这整条链路的一致性
- v9 基本把这条链路完整跑通了，也为后续继续扩展系统日历能力打下了结构基础
