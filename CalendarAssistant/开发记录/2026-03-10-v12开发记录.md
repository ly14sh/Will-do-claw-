# 2026-03-10 开发记录 (v12版本)

## 需求背景

希望新增“手动上传图片 -> OCR -> 把文字发给 AI -> AI 输出 JSON -> 自动创建日程”的入口，并且入口同时覆盖：

- 主页：FAB 菜单里新增图片导入按钮
- 悬浮窗：输入框发送按钮旁新增图片导入按钮

同时，悬浮窗的交互目标偏“通知中心”风格：

- 图片选择器弹出时悬浮窗隐藏，选择器关闭后悬浮窗立刻恢复，并在输入框内展示 Loading
- 卡片支持左滑保持态 + 长左滑直接执行动作 + 露出编辑按钮
- 编辑不跳回 App（避免打断），尽量在悬浮窗卡片展开区域直接编辑并保存

## 一、本次实现范围

### 1) 手动图片导入（主页）

- 在 `HomePage` 的 FAB 展开菜单中加入“识别图片”按钮
- 选图后：拷贝到 `filesDir/event_screenshots/UPLOAD_*.jpg` -> 下采样解码 -> ML Kit OCR -> 复用 `parseUserText()` -> 新建 `MyEvent` 并写入仓库
- 识别过程中通过 Dialog 展示 Loading

### 2) 手动图片导入（悬浮窗）

- 在悬浮窗输入框 trailingIcon 中加入图片按钮，紧挨发送按钮
- 选择器弹出时隐藏悬浮窗；选择器消失时立即恢复悬浮窗，并在输入框里显示圈圈
- 图片识别与落库完成后提示“已添加/未识别到有效日程/失败”

### 3) 悬浮窗内容优化

- 归档事件不在悬浮窗里显示（`archivedAt != null` 直接过滤）

### 4) 悬浮窗卡片交互升级（通知中心风格）

- 左滑“保持态”：松手后停在揭示位，露出右侧操作区
- 长左滑：直接触发动作（完成/撤销/检票等）
- 操作区增加“编辑”按钮（在动作按钮右侧）

### 5) 悬浮窗内编辑（不跳转）

- 点“编辑”后：卡片连贯展开，进入编辑态（标题/开始日期时间/结束日期时间/地点/备注）
- 日期/时间选择器复用既有滚轮组件（不离开悬浮窗）
- 保存直接调用 `repository.updateEvent()`

## 二、关键设计决策

### 1) OCR -> AI 复用“文本发送链路”

本次手动图片导入不走 `analyzeImage()` 的“布局重构 + 多场景并发识别”链路，而是：

1. `RecognitionProcessor.recognizeText(bitmap)` 拿到 OCR 原始文本
2. 复用 `RecognitionProcessor.parseUserText(text, settings)` 产出单条 `CalendarEventData`
3. 转成 `MyEvent` 并落库

这样改动面最小、逻辑复用最高，且和“输入框里发送文本”的体验一致。

### 2) 悬浮窗隐藏/恢复的坑与修复

最初用 `windowManager.removeView()/addView()` 来隐藏/恢复悬浮窗，部分设备上会出现：

- 视图看似还在但触摸失效
- Compose 不再重组，列表不更新

最终改为“不断开 View，仅禁用触摸并隐藏”：

- 隐藏：`updateViewLayout` + `FLAG_NOT_TOUCHABLE | FLAG_NOT_FOCUSABLE` + `view.visibility = GONE`
- 恢复：恢复原 flags + `view.visibility = VISIBLE`

避免了 ComposeView 反复 add/remove 引发的状态异常。

## 四、已知限制 & 后续建议

- 手动图片导入当前走“单条事件”解析（复用 `parseUserText`）；如果需要一次识别多条事件/取件码聚合，可切换到 `analyzeImage()` 并复用 `saveEventsLocally` 的落库策略。
- 悬浮窗内编辑目前对 `course/recurring` 做了保护，避免在悬浮窗里处理复杂的重复逻辑；后续可考虑：
  - 对重复实例编辑复用主 App 的“脱离系列”逻辑
  - 或在悬浮窗只提供“跳转编辑/只读”策略
