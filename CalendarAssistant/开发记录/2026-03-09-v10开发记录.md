# 2026-03-09 开发记录 (v10版本)

## 需求背景

v10 这一轮开发没有再围绕识别本身扩功能，而是集中解决了两个更影响真实使用体验的问题：

1. **设置侧边栏只能在主页打开，进入二级设置页后几乎失去导航能力**
2. **取件码事件参与日历同步后，`tag` 会出现 `pickup -> general -> pickup` 的来回跳变**

这两个问题表面上分别属于“设置页交互”和“日历同步数据一致性”，但它们都指向同一个更底层的目标：

- **让页面导航行为稳定、连续、可预期**
- **让事件数据在 App <-> 系统日历之间 round-trip 后仍保持语义不丢失**

因此，v10 的重点不是单点修一个 bug，而是对：

- 设置区内部导航壳层
- 侧边栏手势与动画模型
- 返回行为统一
- 系统日历元数据同步策略

做了一轮完整收束。

---

## 一、本轮阶段目标

本轮的阶段目标最终收敛为四项：

- **让任意设置页都能直接滑出侧边栏**
- **让设置区内部切页动画与主页导航风格统一**
- **让系统返回与左上角返回完全走同一套历史规则**
- **让事件 `tag` 能真正同步到系统日历并安全读回**

在开发后半段，又追加了一个体验型目标：

- **让侧边栏内容本身支持上下滚动，避免入口过多时被截断**

## 二、问题定位与最终判断

### 1. 为什么侧边栏只能在主页打开

最初只有主页外层包裹了 `PushSlideLayout`，而设置二级页使用的是普通 `Scaffold`。

这意味着：

- 在主页时，侧边栏手势和遮罩都存在
- 一旦跳进设置详情页，页面树里就不再有抽屉容器
- 因而无法继续左/右滑呼出侧边栏

本质上，这是**导航壳层缺失**，不是单纯手势没注册。

### 2. 为什么早期修法会带来一连串动画问题

最开始尝试过用本地状态直接切换设置内容页，这虽然能快速让侧边栏出现在二级页里，但很快暴露出一组连锁问题：

- 切页时右侧内容会先瞬间变成新页面，再播放收起侧边栏动画
- 返回时经常是硬切，没有像主页一样的推挤式转场
- 返回主页时，布局边界外的侧边栏会在屏幕右半边短暂闪现
- 系统返回和左上角返回不总是走同一条逻辑

最终判断是：

- **设置区内部不能继续靠“手搓页面替换”模拟导航**
- **应该直接引入嵌套 `NavHost`，复用与主页同类型的转场逻辑**

### 3. 为什么系统返回会“有时回到上一个页面”

设置区在重构早期同时存在两套历史：

- 一套是自定义的“最多 3 个、不重复”的设置历史
- 一套是内部 `NavHost` 自己的 back stack

如果系统返回优先被内部导航默认消费，就会偶发回到内部真实上一页，而不是我们想要的“受控历史页”。

因此后续修正为：

- 左上角返回按钮和系统返回都统一走 `handleBackNavigation()`
- 自定义历史优先，历史为空时才真正退出设置区回主页

### 4. 为什么取件码 `tag` 会变成 `general`

问题根因最终确认在**日历同步 round-trip 丢失了 tag 元数据**：

1. OCR / 取件码识别创建本地事件时，`tag = pickup` 是正确的
2. 正向同步写入系统日历时，只写了标准字段：标题、时间、地点、备注等
3. 系统日历中并没有保存 `tag`
4. 反向同步读回系统事件时，`CalendarEventMapper` 无法恢复原始 `tag`
5. 结果映射回 `MyEvent` 时退回默认 `general`

而之所以又会“过一会儿变回 `pickup`”，是因为本地还有一层旧校准逻辑：

- 如果描述里包含 `【取件】`，启动或重载时会重新识别为 `pickup`

于是就形成了表面上的：

- **系统日历同步回来时降级成 `general`**
- **本地再校准或后续链路刷新时又修正回 `pickup`**

这不是胶囊在改数据，而是**同步层丢语义、本地层再兜底补语义**。

## 三、最终产品与交互决策

### 1. 设置区导航策略

最终确定：

- 设置区本身作为一个独立导航壳层存在
- 主页只负责“进入设置区”
- 设置区内部由独立 `NavHost` 管理页面切换

### 2. 侧边栏行为规则

确定为：

- 任意设置页都可以滑出侧边栏
- 不再限制成只有边缘热区才能打开
- 侧边栏打开后仍然支持遮罩点击关闭和滑动关闭

原因很明确：

- 全页可呼出更符合此前主页已有的交互预期
- 边缘热区方案虽然可减少误触，但明显损伤易用性

### 3. 设置区历史规则

最终采用：

- **最多记录 3 个历史页**
- **不允许重复页**
- **当前页不计入历史容量**

行为表现为：

- 侧边栏切到当前页：只关闭侧边栏，不新增历史
- 侧边栏切到历史里已有的页：先去重，再把当前页压入历史末尾
- 系统返回 / 左上角返回：优先返回最近历史页
- 历史空了之后：才返回主页

### 4. 动画策略

最终确定：

- 设置区内部页面切换使用与主页一致的左右滑动转场
- 前进：**右入左出**
- 返回：**左入右出**

并且抽出统一转场函数，避免主页和设置区后续继续分叉。

### 5. `tag` 同步策略

最终没有采用“取件码不同步到系统日历”的简化方案，而是采用更完整的做法：

- **允许 `pickup` 事件正常同步到系统日历**
- **同时把 `tag` 一并作为扩展元数据写入系统日历**
- **反向同步时再把 `tag` 读回 App**

这样能保证：

- 系统日历中仍然有这条事件
- App 内部也不会丢失它的“取件码语义”

## 四、设置区导航与侧边栏重构

### 1. `SettingsDetailScreen` 升级为设置壳层

`SettingsDetailScreen.kt` 在本轮被重构为真正的设置导航容器，主要职责变为：

- 接收侧边栏导航动作
- 承载设置区内部 `NavHost`
- 维护自定义历史栈
- 统一处理返回逻辑
- 为所有设置子页提供统一的顶部返回行为

这样设置区终于从“很多独立页面”变成了“一个完整的二级导航系统”。

### 2. 嵌套 `NavHost` 替代手搓页面切换

本轮新增的关键结构是设置区内部嵌套导航。

设置相关页面被全部收束为 route，包括：

- `AI`
- `Preference`
- `Theme`
- `Archives`
- `Backup`
- `CourseManage`
- `TimeTableManage`
- `SemesterConfig`
- `Laboratory`

这让设置区内部页面切换终于具备：

- 正式的 enter / exit / popEnter / popExit 动画
- 与主页一致的转场方向
- 更容易维护的页面生命周期

### 3. 历史上限与去重逻辑落地

为了满足“最多 3 页、不重复”的需求，内部导航并没有直接完全依赖 `NavController` 默认栈，而是额外保留了一份受控历史。

这一层受控历史解决了两个问题：

- 默认导航栈无法天然表达“最多 3 页”的产品规则
- 默认导航栈也不满足“重复页去重”的要求

### 4. 系统返回与左上角返回统一

重构后，系统返回和左上角返回都统一走同一个 `handleBackNavigation()`：

- 如果侧边栏开着，先关闭侧边栏
- 否则如果有历史页，返回历史页
- 否则退出设置区回主页

这样彻底解决了此前“左上角正常、系统返回偶发走错历史”的不一致问题。

## 五、抽屉动画与视觉问题修复

### 1. 右半边闪现侧边栏问题

在返回主页时，曾出现“屏幕右半边短暂闪出侧边栏”的问题。

根因是：

- 侧边栏本身放在内容左侧并通过负位移隐藏
- 但容器没有裁剪边界
- 路由切场时整个布局整体移动，隐藏在边界外的内容被短暂带入可视区

最终修复方式是：

- 为抽屉外层和内容层都加上 `clipToBounds()`

### 2. 动画时长不统一问题

主页和设置页在早期使用了不同的转场时长，导致观感不一致。

最终通过抽取 `AppNavTransitions.kt` 统一为一套配置：

- `navForwardEnterTransition()`
- `navForwardExitTransition()`
- `navBackwardEnterTransition()`
- `navBackwardExitTransition()`

这让：

- 主页 <-> 设置区
- 设置区内部页面之间

都能共用同一套动画节奏和方向。

## 六、侧边栏本身的可滚动性补齐

随着设置入口逐渐增多，侧边栏开始出现内容可能超出屏幕高度的问题。

最终补上的策略非常直接：

- 让 `SettingsSidebar` 主 `Column` 支持 `verticalScroll`
- 去掉原来依赖 `weight(1f)` 撑底的写法
- 改为在底部保留固定高度留白

结果是：

- 侧边栏横向负责抽屉开合
- 侧边栏内部纵向负责内容滚动

两套手势不再互相冲突。

## 七、系统日历 `tag` round-trip 修复

### 1. 旧问题的本质

此前系统日历同步只保存了“标准事件字段”，没有保存 App 专属语义。

因此系统日历对 App 来说只知道：

- 这是一个标题叫 xxx 的事件
- 在什么时间、什么地点发生

但不知道：

- 这是普通日程、取件码、列车还是打车提醒

这导致一旦从系统日历读回，就只能退回普通事件语义。

### 2. 元数据存储方案

本轮在 `CalendarManager` 中补上了系统日历扩展元数据写入逻辑。

当前写入三项：

- `app_id`
- `event_type`
- `tag`

这样一来，系统日历事件不再只是“显示字段同步”，而是具备了最基本的 App 语义回写能力。

### 3. 反向同步读取方案

查询系统日历事件后，现在会额外批量读取这些扩展属性，并挂回 `SystemEventInfo`。

随后在 `CalendarEventMapper` 中：

- 优先使用元数据中的 `tag`
- 元数据缺失时，再回退到描述关键字推断

这保证了两类场景都能兼容：

- **新同步出来的事件**：直接走元数据恢复，最准确
- **旧历史事件**：即便没有元数据，也能通过描述做兼容恢复

### 4. 防止被错误降级为 `general`

除了写入 / 读取元数据，本轮还增加了一层 merge 防火墙。

在 `AppRepository.mergeIncomingCalendarEvent()` 中：

- 如果本地事件已经是 `pickup/train/taxi`
- 而反向同步传回来的却只是 `general`

就优先保留本地已有 tag，不允许无意义降级。

这个保护的意义在于：

- 给旧数据和兼容场景一个缓冲期
- 避免在 metadata 还没补齐的过渡阶段继续出现 `pickup -> general` 的倒退

### 5. 对旧数据的影响

这轮修复并不要求用户手动迁移历史数据。

因为：

- 已经存在的老事件会在下一次正向同步时补写元数据
- 旧事件即便暂时没有 metadata，也仍然可以依赖描述关键字做回退识别

因此整体升级路径是平滑的。

## 十、后续建议

虽然本轮的核心问题已经解决，但从可维护性角度看，后续还可以继续做两件事：

### 1. 为日历同步增加更明确的调试日志

建议在正向 / 反向同步日志中直接打印：

- 本地 `tag`
- 写入系统日历的 `tag`
- 从系统日历读回的 `tag`

这样后续如果再出现语义错乱，能更快定位到底丢在写入、读取还是 merge。

### 2. 为系统日历 metadata 兼容性做一次设备侧回归

理论上 `ExtendedProperties` 方案更正确，但不同 ROM / 不同日历实现对扩展属性的支持稳定性可能存在差异。

建议后续在这些场景做一次回归：

- 原生 Android 日历
- Flyme / One UI 等定制系统
- 不同账号类型的日历（本地 / Google / 交换等）

如果某些设备对扩展属性支持不稳定，再决定是否增加 description 级别的冗余兜底。

## 十一、总结

v10 的本质，不是继续给页面加入口，也不是再堆一个同步补丁，而是把两条此前已经开始变复杂的主线重新收束：

- **设置区从“散落页面”收束成“可维护的二级导航系统”**
- **日历同步从“字段同步”升级成“带语义的 round-trip 同步”**

前者解决的是操作流畅度和导航一致性，后者解决的是跨系统数据语义丢失问题。

这两块稳定下来之后，后续无论继续扩充设置页，还是继续增强识别 / 胶囊 / 取件码链路，都会更稳。
