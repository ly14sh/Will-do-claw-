# Will do v1.5.0 架构重构与优化计划

## 一、背景与问题

### 当前问题 (v1.4.x)

1. **OCR 识别重复 bug**：截图识别时同时调用两个 AI（schedule + pickup），导致同一张图可能生成"普通日程"和"临时事件"两份结果
2. **类型命名混乱**：`temp` 语义不明确，与 `event`/`course` 逻辑边界模糊

---

## 二、核心目标

| 目标 | 描述 |
|------|------|
| 修复 bug | 解决 OCR 双重生成导致的重复日程问题 |
| 架构优化 | `temp` → `pickup`，保持逻辑隔离（不同步、有宽限期、可聚合） |

---

## 三、实施计划

### 阶段一：数据模型标准化

#### 1.1 常量定义 (MyEvent.kt)

```kotlin
companion object {
    const val TYPE_EVENT = "event"   // 普通日程、会议、课程、火车、网约车
    const val TYPE_PICKUP = "pickup" // 取件、取餐、核销码
    const val TYPE_COURSE = "course" // 课程
}
```

#### 1.2 数据迁移 (AppRepository.kt)

启动时一次性迁移：将 JSON 中所有 `temp` 更新为 `pickup`

```kotlin
init {
    scope.launch {
        val events = eventSource.getAllEvents()
        val needMigration = events.any { it.eventType == "temp" }
        if (needMigration) {
            val migratedEvents = events.map {
                if (it.eventType == "temp") it.copy(eventType = TYPE_PICKUP) else it
            }
            eventSource.saveAll(migratedEvents)
            Log.i("Migration", "已迁移 ${events.size} 条旧数据")
        }
    }
}
```

### 阶段二：AI Prompt 统一化

#### 2.1 统一 Prompt 设计

**核心策略**：
- 单一 Prompt，同时识别所有类型
- JSON Schema 约束输出格式
- 多对象列表支持共存场景
- 互斥规则避免分类错误

**关键指令**（添加到 Prompt 注意事项）：
> 如果文本中同时包含取件码和普通日程（例如一张图里既有快递短信又有会议通知），请务必在 events 列表中生成多个独立对象，严禁遗漏任何一方。

#### 2.2 输出格式

```json
{
  "events": [
    {
      "type": "pickup",
      "title": "取件码 5566",
      "description": "【取件码】5566",
      "startTime": "14:30",
      "endTime": "15:30"
    },
    {
      "type": "event",
      "title": "团队会议",
      "description": "【用车】特斯拉 | 粤B12345",
      "startTime": "15:00",
      "endTime": "16:00"
    }
  ]
}
```

#### 2.3 微格式规范

| 类型 | 微格式 | 示例 |
|------|--------|------|
| 火车 | `【列车】车次\|检票口\|座位号` | `【列车】G1024\|8A\|15A` |
| 网约车 | `【用车】车型\|车牌号` | `【用车】特斯拉\|粤B12345` |
| 取件码 | `【取件码】号码` | `【取件码】5566` |
