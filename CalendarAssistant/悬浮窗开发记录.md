## 2026-03-01 音量键拦截逻辑重写 (v2.0)

### 问题背景

原方案 v1.0：使用 HandlerThread + postDelayed 异步触发
- 多次使用 System.currentTimeMillis()（时间可能跳跃）
- 手动保存/恢复音量（有竞态风险）
- 代码复杂，状态变量多

**问题**：
1. 打开悬浮窗后音量仍持续降低到 0
2. 打开状态栏后音量狂减

### 解决方案：参考 KeyMapper 简化方案

核心思路（参考 KeyMapper 的 KeyMapAlgorithm）：
1. 使用 `SystemClock.elapsedRealtime()` 代替 `System.currentTimeMillis()`
2. 在 ACTION_UP 同步判断，不再异步
3. 使用 `repeatCount` 处理重复按键
4. 悬浮窗打开后放行，让系统接管

**v2.0 算法流程**：
```
ACTION_DOWN (首次按下):
  - 如果悬浮窗已显示: 返回 false 放行
  - 如果 repeatCount > 0: 返回 true 拦截（长按期间的重复事件）
  - 记录按下时间，返回 false 让系统处理

ACTION_UP:
  - 计算按压时长
  - >= 500ms (长按): 触发悬浮窗，返回 true 拦截
  - < 500ms (短按): 返回 false 放行
```

**关键代码**：
```kotlin
companion object {
    private const val LONG_PRESS_THRESHOLD = 500L
}

private var volumeDownTime: Long = 0

override fun onKeyEvent(event: KeyEvent): Boolean {
    if (event.keyCode != KeyEvent.KEYCODE_VOLUME_DOWN) {
        return super.onKeyEvent(event)
    }

    // 如果悬浮窗已显示，放行让系统处理
    if (FloatingScheduleService.isShowing) {
        return false
    }

    // 处理重复 DOWN 事件（长按时持续发送）
    if (event.action == KeyEvent.ACTION_DOWN && event.repeatCount > 0) {
        return true
    }

    when (event.action) {
        KeyEvent.ACTION_DOWN -> {
            volumeDownTime = SystemClock.elapsedRealtime()
            return false
        }

        KeyEvent.ACTION_UP -> {
            val pressDuration = if (volumeDownTime > 0) {
                SystemClock.elapsedRealtime() - volumeDownTime
            } else {
                0
            }

            volumeDownTime = 0

            return if (pressDuration >= LONG_PRESS_THRESHOLD) {
                startFloatingService()
                true  // 长按，拦截
            } else {
                false  // 短按，放行
            }
        }
    }
    return super.onKeyEvent(event)
}
```

## 2026-03-02 长侧滑手势方案 (v3.0)

### 问题背景

v2.0 音量键方案存在问题：
- 长按音量下时，音量会被降低到 0
- 短按无法降低音量
- 可能影响电源+音量截屏组合键

### 解决方案：长侧滑手势

用户从屏幕左/右边缘开始滑动，超过屏幕一半距离时触发悬浮窗：
- 从左边滑动 → 悬浮窗日程卡片显示在左边
- 从右边滑动 → 悬浮窗日程卡片显示在右边
- 悬浮窗关闭后 → 立即恢复手势监听

### v3.0 方案流程

```
用户从屏幕边缘开始滑动
        ↓
滑动距离 > 屏幕宽度 / 2
        ↓
显示悬浮窗 + 停止手势监听
        ↓
悬浮窗关闭 → 恢复手势监听
```

```kotlin
companion object {
    private const val EDGE_THRESHOLD = 50f  // 边缘检测阈值
}

private var isListeningGesture = true
private var screenWidth = 0
private var gestureStartX = 0f

override fun onServiceConnected() {
    super.onServiceConnected()
    val display = windowManager.defaultDisplay
    val metrics = DisplayMetrics()
    display.getRealMetrics(metrics)
    screenWidth = metrics.widthPixels
}

// 新增手势检测方法
private fun handleGesture(startX: Float, deltaX: Float): Boolean {
    if (!isListeningGesture || FloatingScheduleService.isShowing) {
        return false
    }
    
    val threshold = screenWidth / 2
    
    // 从左边滑动超过阈值
    if (gestureStartX < EDGE_THRESHOLD && deltaX > threshold) {
        startFloatingService(fromLeft = true)
        return true
    }
    
    // 从右边滑动超过阈值
    if (gestureStartX > screenWidth - EDGE_THRESHOLD && deltaX < -threshold) {
        startFloatingService(fromLeft = false)
        return true
    }
    
    return false
}

private fun startFloatingService(fromLeft: Boolean) {
    isListeningGesture = false
    val intent = Intent(this, FloatingScheduleService::class.java).apply {
        putExtra("from_left", fromLeft)
        addFlags(Intent.FLAG_ACTIVITY_NEW_TASK)
    }
    startService(intent)
}

fun resumeGestureListening() {
    isListeningGesture = true
}
```

## 2026-03-02 日程左滑操作功能 (v4.0)

### 功能概述

在悬浮窗中对日程卡片进行左滑操作，模拟胶囊/通知中的"已完成"按钮：

- 左滑触发操作 → 震动反馈
- 首次左滑：显示提示泡泡"再次左滑XXX"
- 再次左滑：执行操作
- 1分钟内可点击撤销按钮取消操作

### 操作类型判断

| 条件 | 操作 | 状态标签 |
|------|------|----------|
| event.eventType == "pickup" | 已取件 | 已取件 |
| event.tag == "train" | 已检票 | 已检票 |
| event.tag == "taxi" | 已用车 | 已用车 |
| 其他 | 已完成 | 已完成 |

### 数据模型修改

```kotlin
val originalEndTime: String? = null,         // 原始结束时间（用于撤销恢复）

```

   - 保存 originalEndDate/originalEndTime

5. `undoCheckInTransport(id: String): Boolean` - 撤销检票（检查1分钟内）

v4.0 版本的"再次左滑"提示泡泡在实际使用中显得多余，用户反馈：
1. 右上角已有状态标签显示变化
2. 震动反馈已提供操作确认
3. 二次确认增加了操作复杂度

### 1. 删除提示泡泡

**删除内容：**
- `showSwipeHint` 状态变量
- `LaunchedEffect` 提示泡泡定时消失逻辑
- 左滑提示泡泡 UI 代码

**简化左滑逻辑：**
- 首次左滑直接执行操作（去掉二次确认）
- 震动后立即触发

### 2. 添加卡片滑动动画

**实现方式：**
- 使用 `animateFloatAsState` + `spring` 实现平滑跟随手指移动
- 松开后自动弹回

```kotlin
var offsetX by remember { mutableFloatStateOf(0f) }
var isDragging by remember { mutableStateOf(false) }

val animatedOffset by animateFloatAsState(
    targetValue = if (isDragging) offsetX else 0f,
    animationSpec = spring(
        dampingRatio = Spring.DampingRatioMediumBouncy,
        stiffness = Spring.StiffnessLow
    ),
    label = "cardOffset"
)

// 手势处理
detectHorizontalDragGestures(
    onDragStart = { isDragging = true },
    onDragEnd = { ... },
    onHorizontalDrag = { change, dragAmount ->
        offsetX = (offsetX + dragAmount).coerceIn(-200f, 0f)
    }
)

// 卡片应用偏移
Surface(
    modifier = Modifier.offset(x = animatedOffset.dp)
) { ... }
```

### 3. 添加背景图标

**触发条件：** 滑动距离超过 30dp

**图标类型：**
| 条件 | 图标 | 文字 | 颜色 |
|------|------|------|------|
| event.tag == "train" | Train | 已检票 | #4CAF50 |
| event.eventType == "pickup" | Checkroom | 已取件 | #2196F3 |
| event.tag == "taxi" | DirectionsCar | 已用车 | #FF9800 |
| 其他 | CheckCircle | 已已完成 | #4CAF50 |

**透明度变化：**
```kotlin
alpha = (-animatedOffset / 100f).coerceIn(0.3f, 1f)
// 滑动越多越清晰，最少 30% 透明度
```

## 2026-03-03 UI优化 + Bug修复 + Prompt增强

### 1. UI 优化 (FloatingScheduleScreen.kt)

#### 卡片宽度调整
- 从 300dp 调整为 280dp（短小精悍）

#### 背景图标调整
- padding 从 40dp 调整为 24dp（更靠右贴边）
- 移除背景文字，仅保留图标圆圈

#### 图标更新
| 场景 | 原图标 | 新图标 |
|------|--------|--------|
| train | Train (火车) | ConfirmationNumber (票根) |
| taxi | DirectionsCar (轿车) | LocalTaxi (出租车) |
| pickup | Checkroom (衣架) | ShoppingBag (购物袋) |

### 2. Bug 修复 (FloatingScheduleService.kt)

**问题**：`convertToMyEvent` 方法未传递 tag 字段，导致 AI 返回的 tag 被丢弃

**修复**：添加 `tag = eventData.tag`

```kotlin
return MyEvent(
    // ... 其他字段
    eventType = eventType,
    tag = eventData.tag  // 新增
)
```

### 3. Prompt 优化 (AiPrompts.kt)

**主要改动**：
1. 增加场景判定关键词
   - 场景1：车次/座位号/12306/检票口 → train；车牌/车型/司机/行程单 → taxi
   - 场景2：取件码/提货码/快递单号/取餐号 → pickup
   - 场景3：不符合上述的都 → general

2. 强调 tag 字段为"必须"
   - itemSchema: `"tag": "【必须】general | pickup | train | taxi"`
   - 每个场景明确 "【强制】tag: xxx"

3. 场景3增加负面约束
   - "仅包含'去某地'但无车次/车牌号的，归为 'general'"

4. 优化 reasoning 输出
   - 要求包含：判定场景、时间依据、tag设置理由
