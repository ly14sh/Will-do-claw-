# 第一部分：项目概述

## 1.1 项目简介

Will do（智能日历助手）是一款基于 Android 平台的智能日程管理应用，采用 Jetpack Compose 构建现代化用户界面，应用 MVVM 架构模式实现清晰的数据分层和模块化设计。项目经过多个版本的迭代优化，已形成一套完整的日程管理解决方案。

该应用的核心设计理念是「智能、简洁、实用」，通过 AI 技术和系统深度集成，为用户提供从日程识别创建到提醒通知的完整闭环体验。无论是课程表管理、日常会议安排，还是取件取餐提醒、交通出行通知，应用都能提供智能化的处理方案。

项目名称 "Will do" 体现了积极主动的生活态度，寓意帮助用户更好地规划和管理时间，让每一项待办事项都能得到妥善处理。应用通过简洁直观的界面设计和智能化的功能实现，让日程管理变得轻松高效。

### 技术选型说明

**Jetpack Compose** 是 Google 推荐的现代 Android UI 工具包，它使用声明式编程范式，允许开发者以更直观的方式构建 UI。相比传统的 XML 布局方式，Compose 提供了更强大的动画支持、更灵活的布局系统，以及更简洁的代码风格。在本项目中，100% 的 UI 都采用 Compose 实现，包括复杂的课表视图、手势交互和动画效果。

**Ktor Client** 是一个轻量级的 Kotlin 异步 HTTP 客户端，被选用于 AI API 的调用。相比其他网络库，Ktor 提供了更好的 Kotlin 协程支持和更简洁的 API 设计，非常适合处理 AI 请求这类需要异步处理的场景。Ktor 的内容协商功能与 Kotlinx Serialization 无缝集成，可以自动处理 JSON 的序列化与反序列化。

**ML Kit Text Recognition** 是 Google 提供的机器学习文字识别库，专门针对移动设备进行了优化。支持中文识别是选择该库的关键原因，因为在日程识别场景中，中文文字的提取是核心需求。ML Kit 可以在设备本地完成识别，无需联网，保护用户隐私的同时也提高了响应速度。

**Kotlinx Serialization** 是 Kotlin 官方提供的多平台序列化库，支持 Kotlin 数据类的自动序列化。相比 Gson 等传统 JSON 库，Serialization 提供了更强的类型安全性和更简洁的使用方式，特别适合与 Kotlin 的数据类（data class）配合使用。

### 架构模式：MVVM + Repository

项目采用经典的 MVVM（Model-View-ViewModel）架构，并在此基础上引入了 Repository 模式进行数据抽象。这种架构的优势在于：

- **关注点分离**：UI 层只负责界面展示，业务逻辑集中在 ViewModel 中，数据操作通过 Repository 统一管理
- **响应式数据流**：使用 Kotlin StateFlow 实现数据的响应式更新，UI 自动响应数据变化
- **单例 Repository**：AppRepository 采用单例模式，确保整个应用共享同一数据源，避免数据不一致问题
- **协程支持**：所有耗时操作（网络请求、文件读写）都运行在协程中，不会阻塞主线程

## 1.4 核心功能一览

Will do 应用提供了丰富的日程管理功能，本节将对各项核心功能进行概览性介绍，后续章节会详细展开说明。

### 智能日程识别

应用支持两种智能识别方式：截图识别和文字输入识别。截图识别利用无障碍服务获取屏幕内容，结合 ML Kit 进行 OCR 文字提取，再通过 AI 大模型解析语义信息生成结构化日程数据。文字输入识别则直接接收用户输入的自然语言描述，利用 AI 解析生成日程。

AI 识别采用双 Prompt 并发模式：Schedule Prompt 负责识别普通日程、会议、交通出行等信息，Pickup Prompt 专注于取件码、取餐码等临时事件的识别。两个请求并发执行，通过去重逻辑合并结果，既提高了识别速度，又保证了各类信息的完整性。

### 课程表管理系统

课程表管理是应用的另一核心功能。用户可以添加、编辑、删除课程，支持单双周课程、排除特定日期、临时调课等复杂场景。课程数据以 JSON 格式存储在本地，与普通日程分离管理。课程在运行时会被转换为虚拟日程显示在时间轴上，但不会污染系统日历。

应用支持从外部导入课程表数据，目前兼容醒课表的数据格式。导入功能会自动解析课程信息，包括课程名称、上课地点、授课教师、节次时间等，并支持学期的周次计算。

### 实况胶囊通知

实况胶囊（Live Activity）是 Android 14 引入的新特性，可以在锁屏界面和灵动岛区域显示实时更新的通知内容。应用利用这一特性，在日程即将开始时显示胶囊通知，包含日程标题、剩余时间、快捷操作按钮等信息。

除了原生 Android 实况胶囊，应用还适配了 Flyme 系统的实况通知功能。Flyme 是国内定制系统中最早支持类似功能的厂商，其胶囊实现方式与原生略有不同，应用通过 RemoteViews 自定义布局来兼容这两种显示方式。

胶囊状态管理器（CapsuleStateManager）是整个胶囊功能的核心，它统一计算和管理所有需要显示胶囊的日程，包括普通日程、课程和取件码。管理器采用主动唤醒模式，当检测到需要显示胶囊时会自动启动前台服务。

### 悬浮窗功能

悬浮窗是 v1.5 版本新增的重要功能，允许用户通过长按音量+键呼出悬浮日程窗口。悬浮窗以系统悬浮窗的形式存在，可以覆盖在其他应用之上显示，非常适合在需要快速查看或操作日程时使用。

悬浮窗提供了智能标题显示功能：根据日程类型动态展示关键信息。火车票会显示检票口或座位号，网约车会显示车牌号，取件码会直接显示取件号码。悬浮窗还支持左滑操作，用户可以通过滑动完成已取件、已检票等快捷操作。

### 日历同步

应用支持与系统日历进行双向同步：可以将本地日程同步到系统日历，也可以在应用启动时从系统日历导入日程。同步功能采用去重机制，避免重复创建相同日程。用户可以在设置中选择同步方向（双向/仅上传/仅下载），并排除特定类型的日程（如取件码）。

日历同步管理器（CalendarSyncManager）负责处理所有同步相关的逻辑，包括日历账户的查询、事件的创建与更新、冲突解决策略等。同步操作运行在后台协程中，不会影响应用的响应速度。

### 其他功能

除了上述核心功能外，应用还提供了多项实用特性：每日日程汇总通知（在早间和晚间推送当日日程概要）、智能推荐（基于历史数据推荐合适的日程安排）、自动归档（过期日程自动归档以减少干扰）、网速胶囊（在胶囊中显示当前网络速度）等。这些功能大多可以通过设置进行开关控制，用户可以根据自己的需求进行定制。

### 各层职责说明

**UI 层** 负责所有用户可见的内容，包括页面布局、组件渲染和用户交互响应。UI 层不直接处理业务逻辑，而是通过观察 ViewModel 中的 StateFlow 来获取数据变化，并据此更新界面。所有用户操作都会被转发到 ViewModel 进行处理。

**业务逻辑层（core）** 包含了应用的核心功能实现，如 AI 识别处理、日历操作封装、胶囊状态计算、课程管理等。这一层是应用的大脑，负责所有的决策和计算逻辑。业务逻辑层通过调用数据层接口获取数据，处理后再将结果暴露给 UI 层。

**数据层** 负责所有数据的持久化和读取操作。数据层采用 Repository 模式，对外提供统一的数据访问接口，屏蔽了底层存储细节。数据层内部使用 JSON 文件作为存储介质，通过多个 DataSource 类分别管理不同类型的数据。

**服务层** 处理需要在后台长期运行的任务，如悬浮窗显示、胶囊通知、闹钟提醒等。服务层通过系统服务（Service）和广播接收器（BroadcastReceiver）与操作系统进行交互，实现各种系统级别的功能。

## 2.2 MVVM 架构实现

### ViewModel 与 StateFlow

项目中主要使用两个 ViewModel：MainViewModel 负责主页相关的业务逻辑，SettingsViewModel 负责设置页面的逻辑。每个 ViewModel 都通过 StateFlow 向 UI 暴露状态数据。

```kotlin
class MainViewModel(
    private val repository: AppRepository
) : ViewModel() {
    
    // 暴露给 UI 的状态数据（只读）
    val selectedDate: StateFlow<LocalDate> = _selectedDate.asStateFlow()
    val eventsForSelectedDate: StateFlow<List<MyEvent>> = _eventsForSelectedDate.asStateFlow()
    val coursesForSelectedDate: StateFlow<List<Course>> = _coursesForSelectedDate.asStateFlow()
    
    // UI 事件处理
    fun onDateSelected(date: LocalDate) {
        _selectedDate.value = date
    }
    
    fun onEventCompleted(eventId: String) {
        viewModelScope.launch {
            repository.completeScheduleEvent(eventId)
        }
    }
}
```

StateFlow 是 Kotlin 协程库提供的响应式数据容器，与 LiveData 类似但更加强大。StateFlow 始终持有最新的数据值，当数据变化时会自动通知所有观察者。与 LiveData 不同的是，StateFlow 不需要.lifecycleScope 来自动取消订阅，也不局限于在 Compose 中使用。

### Repository 单例模式

AppRepository 是整个应用的数据枢纽，采用单例模式确保全局唯一：

```kotlin
class AppRepository private constructor(private val context: Context) {
    
    // 内部可变的状态
    private val _events = MutableStateFlow<List<MyEvent>>(emptyList())
    private val _courses = MutableStateFlow<List<Course>>(emptyList())
    private val _settings = MutableStateFlow(MySettings())
    
    // 暴露给外部的只读状态
    val events: StateFlow<List<MyEvent>> = _events.asStateFlow()
    val courses: StateFlow<List<Course>> = _courses.asStateFlow()
    val settings: StateFlow<MySettings> = _settings.asStateFlow()
    
    companion object {
        @Volatile
        private var INSTANCE: AppRepository? = null
        
        fun getInstance(context: Context): AppRepository {
            return INSTANCE ?: synchronized(this) {
                INSTANCE ?: AppRepository(context.applicationContext).also { INSTANCE = it }
            }
        }
    }
}
```

单例模式保证了整个应用使用同一份数据，避免了因多个数据实例导致的同步问题。同时，所有数据操作都通过 Repository 进行，使得数据流清晰可控，便于调试和优化。

## 2.3 胶囊状态管理架构

胶囊状态管理是应用的核心创新点之一，它实现了一种「主动唤醒」的设计模式：将胶囊显示的计算逻辑与前台服务分离，由 CapsuleStateManager 统一计算状态，当状态变为 Active 时自动启动 CapsuleService。

### CapsuleStateManager 设计

```kotlin
class CapsuleStateManager(
    private val repository: AppRepository,
    private val appScope: CoroutineScope,
    private val context: Context
) {
    
    // 胶囊 UI 状态 - StateFlow
    val uiState: StateFlow<CapsuleUiState> = createCapsuleStateFlow()
    
    init {
        // 主动唤醒机制：监听 uiState，当变为 Active 时启动 Service
        startServiceWakeup()
    }
    
    // 合并多个数据源：events + courses + settings + ticker(每分钟)
    private fun createCapsuleStateFlow(): StateFlow<CapsuleUiState> {
        val tickerFlow = flow {
            emit(System.currentTimeMillis()) // 立即发射
            while (true) {
                delay(60_000) // 每分钟更新
                emit(System.currentTimeMillis())
            }
        }
        
        return combine(
            repository.events,
            repository.courses,
            repository.settings,
            tickerFlow
        ) { events, courses, settings, _ ->
            computeCapsuleState(events, courses, settings)
        }.stateIn(...)
    }
}
```

这种设计的核心优势在于：

1. **计算与显示分离**：CapsuleStateManager 只负责状态计算，CapsuleService 只负责界面显示，职责清晰
2. **响应式更新**：利用 combine 操作符将多个数据源合并，任何一个数据源变化都会触发重新计算
3. **主动唤醒**：通过 collect 监听状态变化，当需要显示胶囊时自动启动服务，不再需要时自动停止
4. **强制刷新**：提供 forceRefresh() 方法，可以在特定操作后立即触发状态更新

### CapsuleService Dumb Service 模式

CapsuleService 采用「Dumb Service」（哑服务）模式运行，它本身不进行任何业务逻辑计算，只是简单地监听状态变化并更新通知：

```kotlin
class CapsuleService : Service() {
    
    private val repository by lazy { AppRepository.getInstance(applicationContext) }
    private val capsuleManager by lazy { 
        CapsuleStateManager(repository, serviceScope, applicationContext) 
    }
    
    override fun onCreate() {
        super.onCreate()
        // 监听胶囊状态，状态变化时更新通知
        serviceScope.launch {
            capsuleManager.uiState.collect { state ->
                when (state) {
                    is CapsuleUiState.Active -> showCapsuleNotification(state)
                    is CapsuleUiState.None -> cancelAllNotifications()
                }
            }
        }
    }
}
```

这种模式避免了服务本身维护复杂状态，所有状态都由 CapsuleStateManager 统一管理，服务只是一个执行者。

# 第四部分：核心功能详解

## 4.1 AI 智能识别

AI 智能识别是应用最具特色的功能之一，它能够从截屏或文本中自动提取日程信息。用户无需手动填写繁琐的表单，只需要提供相关信息，AI 就能完成解析和创建。

### 4.1.1 识别流程架构

### 4.1.2 双 Prompt 并发模式

AI 识别采用双 Prompt 并发模式是 v1.5 版本的重要优化。在此之前，应用使用单一 Prompt 要求 AI 同时识别所有类型的日程，但这种设计存在两个问题：Prompt 过长导致 AI 响应变慢，且某些边界情况的识别准确率不高。

新的设计将识别任务拆分为两个独立的 Prompt：

**Schedule Prompt（普通日程）**

```
你是一个日程识别助手。从以下文本中提取日程信息。

要求：
1. 识别会议、约会、出行等普通日程
2. 如果文本中有火车票信息，格式为【列车】车次|检票口|座位号
3. 如果文本中有网约车信息，格式为【用车】车型|车牌号
4. 只返回普通日程，不要返回取件码信息
5. tag 字段：general=普通日程, train=火车, taxi=网约车

返回 JSON 格式：
{
  "events": [
    {
      "title": "日程标题",
      "description": "详细描述或交通微格式",
      "startTime": "HH:mm",
      "endTime": "HH:mm",
      "tag": "general|train|taxi"
    }
  ]
}
```

**Pickup Prompt（取件码）**

```
你是一个取件码识别助手。从以下文本中提取取件码信息。

要求：
1. 识别快递取件码、取餐码、核销码等信息
2. 格式统一为【取件】号码|品牌|位置（位置可选）
3. tag 字段统一为 pickup
4. 只返回取件码信息，不要返回普通日程

返回 JSON 格式：
{
  "events": [
    {
      "title": "取件码 XXX",
      "description": "【取件】号码|品牌|位置",
      "tag": "pickup"
    }
  ]
}
```

### 4.1.3 结果合并与去重

由于 Schedule Prompt 和 Pickup Prompt 是并发执行的，可能会出现重复识别的情况。例如，一条同时包含取件码和会议通知的文本，两个 Prompt 可能都会尝试识别。

去重逻辑基于 title 和 startTime 的组合进行分组，优先保留 tag 为 pickup 的结果：

```kotlin
val allEvents = scheduleEvents + pickupEvents
val finalEvents = allEvents
    .groupBy { "${it.title}|${it.startTime}" }
    .map { (_, events) ->
        events.maxByOrNull { if (it.tag == "pickup") 1 else 0 }!!
    }
```

### 4.1.4 微格式规范

为了在日程中携带更多结构化信息，应用定义了以下微格式规范：

| 类型 | 格式 | 示例 |
|------|------|------|
| 火车票 | `【列车】车次\|检票口\|座位号` | `【列车】G1024\|8A\|15A` |
| 网约车 | `【用车】车型\|车牌号` | `【用车】特斯拉\|鄂A12345` |
| 取件码 | `【取件】号码\|品牌\|位置` | `【取件】5566\|顺丰\|菜鸟驿站` |

这些微格式信息会被胶囊状态管理器解析，用于在胶囊和悬浮窗中显示更直观的动态标题。

### 4.1.5 RecognitionProcessor 实现

图像识别处理器协调整个识别流程：

```kotlin
class RecognitionProcessor(
    private val context: Context,
    private val apiProvider: ApiModelProvider
) {
    // 并发执行两个识别请求
    suspend fun analyzeImage(bitmap: Bitmap, settings: MySettings): List<MyEvent> {
        // 1. ML Kit OCR 提取文字
        val extractedText = extractText(bitmap)
        
        if (extractedText.isBlank()) {
            return emptyList()
        }
        
        // 2. 并发执行两个 AI 请求
        val scheduleDeferred = async { analyzeSchedule(extractedText, settings) }
        val pickupDeferred = async { analyzePickup(extractedText, settings) }
        val scheduleEvents = scheduleDeferred.await()
        val pickupEvents = pickupDeferred.await()
        
        // 3. 合并结果并去重
        return mergeAndDeduplicate(scheduleEvents, pickupEvents)
    }
    
    private fun extractText(bitmap: Bitmap): String {
        val recognizer = TextRecognition.getClient(
            ChineseTextRecognizerOptions.Builder().build()
        )
        // 同步等待 OCR 结果
        val result = recognizer.process(bitmap).await()
        return result.text
    }
    
    private suspend fun analyzeSchedule(text: String, settings: MySettings): List<MyEvent> {
        val prompt = AiPrompts.getSchedulePrompt()
        val response = apiProvider.generate(prompt, text, settings)
        return parseEventsResponse(response)
    }
    
    private suspend fun analyzePickup(text: String, settings: MySettings): List<MyEvent> {
        val prompt = AiPrompts.getPickupPrompt()
        val response = apiProvider.generate(prompt, text, settings)
        return parseEventsResponse(response)
    }
    
    // 按 title+startTime 分组去重，优先保留 tag=pickup 的结果
    private fun mergeAndDeduplicate(
        scheduleEvents: List<MyEvent>,
        pickupEvents: List<MyEvent>
    ): List<MyEvent> {
        val allEvents = scheduleEvents + pickupEvents
        return allEvents
            .groupBy { "${it.title}|${it.startTime}" }
            .map { (_, events) ->
                events.maxByOrNull { if (it.tag == "pickup") 1 else 0 }!!
            }
    }
}
```

### 4.2.3 课程筛选逻辑

1. 当前周在课程的周次范围内：`currentWeek in startWeek..endWeek`
2. 单双周匹配：当前周类型与课程 weekType 一致（weekType=0 表示每周）
3. 星期几匹配：课程设置的 dayOfWeek 等于当前日期的星期
4. 不在排除日期列表中：当前日期不在 excludedDates 中

### 4.2.4 影子课程机制

影子课程（Temp Course）是用于处理单次课程调整的机制。当某次课程临时调课、更换教室或取消时，不需要修改主课程设置，而是创建一个影子课程：

- 影子课程的 isTemp 标记为 true
- 影子课程关联父课程的 parentCourseId
- 原主课程将该日期加入 excludedDates

这种设计的优势在于：学期结束后清理课程时，只需要删除主课程，影子课程会作为附带清理，无需单独处理。

### 4.2.5 课程导入

- 课程名称 → name
- 上课地点 → location
- 授课教师 → teacher
- 星期几 + 节次 → dayOfWeek, startNode, endNode

导入前会进行去重检查，避免重复创建相同的课程。

### 4.2.6 CourseManager 实现

课程管理器负责课程数据与虚拟日程之间的转换：

```kotlin
class CourseManager(
    private val repository: AppRepository
) {
    // 将课程转换为虚拟日程显示
    fun getCoursesForDate(date: LocalDate): List<MyEvent> {
        val courses = repository.courses.value
        val currentWeek = calculateCurrentWeek()
        val dayOfWeek = date.dayOfWeek.value
        
        return courses
            .filter { course -> isCourseActive(course, currentWeek, dayOfWeek, date) }
            .map { course -> convertToEvent(course, date) }
    }
    
    private fun isCourseActive(
        course: Course,
        currentWeek: Int,
        dayOfWeek: Int,
        date: LocalDate
    ): Boolean {
        // 1. 周次范围检查
        if (currentWeek !in course.startWeek..course.endWeek) return false
        
        // 2. 单双周检查
        if (!isWeekTypeMatch(course.weekType, currentWeek)) return false
        
        // 3. 星期几检查
        if (course.dayOfWeek != dayOfWeek) return false
        
        // 4. 排除日期检查
        val dateStr = date.format(DateTimeFormatter.ISO_LOCAL_DATE)
        if (dateStr in course.excludedDates) return false
        
        return true
    }
    
    private fun isWeekTypeMatch(weekType: Int, currentWeek: Int): Boolean {
        return when (weekType) {
            0 -> true  // 每周
            1 -> currentWeek % 2 == 1  // 单周
            2 -> currentWeek % 2 == 0  // 双周
            else -> true
        }
    }
    
    private fun convertToEvent(course: Course, date: LocalDate): MyEvent {
        val timeTable = parseTimeTableJson(repository.settings.value.timeTableJson)
        val startNode = course.startNode
        val endNode = course.endNode
        
        return MyEvent(
            id = "course_${course.id}_${date}",
            title = course.name,
            startDate = date,
            endDate = date,
            startTime = timeTable[startNode].startTime,
            endTime = timeTable[endNode].endTime,
            location = course.location,
            description = course.teacher,
            color = course.color,
            eventType = EventType.COURSE
        )
    }
}
```

## 4.3 实况胶囊

实况胶囊（Live Activity）是 Android 14 引入的新特性，应用利用这一特性在锁屏界面和灵动岛区域显示日程信息。

### 4.3.1 胶囊状态计算

胶囊状态由 CapsuleStateManager 统一计算，计算时会考虑以下因素：

1. **活跃事件**：当前时间在事件开始时间和结束时间范围内的事件
2. **即将开始的事件**：在提前提醒时间范围内（如 30/45/60 分钟）的事件
3. **取件码**：未过期的取件码，支持聚合显示
4. **课程**：即将开始的课程
5. **全局提前提醒设置**：用户设置的提前提醒分钟数

### 4.3.2 胶囊类型

胶囊分为以下几种类型：

| 类型 | 条件 | 显示内容 |
|------|------|----------|
| 日程 | 即将开始或进行中 | 标题 + 剩余时间/进行中 |
| 课程 | 即将开始或进行中 | 课程名 + 地点 + 剩余时间 |
| 取件码 | 未过期 | 取件码号码（可聚合） |
| 火车票 | 即将开始 | 检票口/座位号 + 车次 |
| 网约车 | 即将开始 | 车牌号 + 车型 |

### 4.3.3 取件码聚合

当开启取件码聚合功能时，多个未过期的取件码会被合并显示在一个胶囊中。聚合胶囊会显示取件码的数量，而不是具体的号码详情：

```

```

点击聚合胶囊可以展开查看具体的取件码列表。

**CapsuleStateManager 核心实现**：

```kotlin
class CapsuleStateManager(
    private val repository: AppRepository,
    private val appScope: CoroutineScope,
    private val context: Context
) {
    // 使用 MutableStateFlow(0) 确保 combine 永远不会因为等待信号而卡死
    private val forceRefreshTrigger = MutableStateFlow(0)
    
    // 网速胶囊的动态状态
    private val networkSpeedState = MutableStateFlow<NetworkSpeedMonitor.NetworkSpeed?>(null)
    
    // 合并多个数据源：events + courses + settings + ticker(每分钟)
    val uiState: StateFlow<CapsuleUiState> = createCapsuleStateFlow()
    
    private fun createCapsuleStateFlow(): StateFlow<CapsuleUiState> {
        val tickerFlow = flow {
            emit(System.currentTimeMillis()) // 立即发射
            while (true) {
                delay(60_000) // 每分钟更新
                emit(System.currentTimeMillis())
            }
        }
        
        return combine(
            repository.events,
            repository.courses,
            repository.settings,
            tickerFlow
        ) { events, courses, settings, _ ->
            computeCapsuleState(events, courses, settings)
        }.stateIn(...)
    }
    
    // 强制刷新胶囊状态（同步执行，确保调用后立即生效）
    fun forceRefresh() {
        val newValue = forceRefreshTrigger.value + 1
        forceRefreshTrigger.value = newValue
    }
}
```

```kotlin
fun isWithinAdvanceReminder(event: MyEvent, settings: MySettings): Boolean {
```

**FloatingScheduleService 核心实现**：

```kotlin
class FloatingScheduleService : Service(), LifecycleOwner, SavedStateRegistryOwner {
    
    companion object {
        @Volatile var isShowing: Boolean = false
            private set
    }
    
    private lateinit var windowManager: WindowManager
    private var composeView: ComposeView? = null
    private val serviceScope = CoroutineScope(Dispatchers.Main + SupervisorJob())
    
    // 广播接收器：监听关闭和系统对话框
    private val closeReceiver = object : BroadcastReceiver() {
        override fun onReceive(context: Context, intent: Intent) {
            when (intent.action) {
                TextAccessibilityService.ACTION_CLOSE_FLOATING -> stopSelf()
                Intent.ACTION_CLOSE_SYSTEM_DIALOGS -> {
                    // 监听 Home 键或多任务键，自动关闭悬浮窗
                    val reason = intent.getStringExtra("reason")
                    if (reason == "homekey" || reason == "recentapps") {
                        stopSelf()
                    }
                }
            }
        }
    }
    
    override fun onCreate() {
        super.onCreate()
        isShowing = true
        windowManager = getSystemService(WINDOW_SERVICE) as WindowManager
        
        // 注册广播接收器
        val filter = IntentFilter().apply {
            addAction(TextAccessibilityService.ACTION_CLOSE_FLOATING)
            if (Build.VERSION.SDK_INT < Build.VERSION_CODES.S) {
                addAction(Intent.ACTION_CLOSE_SYSTEM_DIALOGS)
            }
        }
        registerReceiver(closeReceiver, filter)
        
        // 创建悬浮窗布局
        createFloatingWindow()
    }
    
    private fun createFloatingWindow() {
        val params = WindowManager.LayoutParams().apply {
            type = WindowManager.LayoutParams.TYPE_APPLICATION_OVERLAY
            format = PixelFormat.RGBA_8888
            flags = WindowManager.LayoutParams.FLAG_NOT_FOCUSABLE
            // ... 其他布局参数
        }
        
        composeView = ComposeView(context).apply {
            setContent {
                CalendarAssistantTheme {
                    FloatingScheduleScreen(
                        onClose = { stopSelf() }
                    )
                }
            }
        }
        
        windowManager.addView(composeView, params)
    }
}
```

### 4.5.1 同步方向

同步支持三种方向模式：

| 模式 | 说明 |
|------|------|
| 双向同步 | 应用和系统日历互相同步，变更会双向传播 |
| 仅上传 | 只将应用日程同步到系统日历 |
| 仅下载 | 只将系统日历同步到应用 |

### 4.5.2 同步策略

同步过程中采用以下策略：

1. **去重**：基于事件标题和时间的组合判断是否为同一事件
2. **过滤**：可以排除特定类型的日程（如取件码）不同步到日历
3. **冲突解决**：以应用数据为准，系统日历中的对应事件会被更新
4. **单事件同步**：胶囊/悬浮窗中的状态变更会立即同步到日历

### 4.5.3 日历账户

应用支持多日历账户，用户可以选择将日程同步到哪个日历账户：

```kotlin
val calendars = calendarManager.getCalendars()
```

### 4.5.4 CalendarSyncManager 实现

日历同步管理器负责处理所有同步相关的逻辑：

```kotlin
class CalendarSyncManager(
    private val context: Context,
    private val repository: AppRepository
) {
    // 同步方向枚举
    enum class SyncDirection {
        BOTH,       // 双向同步
        UPLOAD,     // 仅上传
        DOWNLOAD    // 仅下载
    }
    
    // 单事件同步（用于胶囊/悬浮窗状态变更后立即同步）
    suspend fun syncEventToCalendar(event: MyEvent): Result<Unit> {
        if (event.eventType == EventType.PICKUP) {
            // 取件码不同步，保持日历干净
            return Result.success(Unit)
        }
        
        return try {
            val calendarId = getOrCreateCalendarId()
            val existingEvent = findExistingEvent(calendarId, event)
            
            if (existingEvent != null) {
                // 更新现有事件
                updateCalendarEvent(existingEvent.id, event)
            } else {
                // 创建新事件
                insertCalendarEvent(calendarId, event)
            }
            Result.success(Unit)
        } catch (e: Exception) {
            Result.failure(e)
        }
    }
    
    // 从系统日历导入事件
    suspend fun importFromCalendar(): List<MyEvent> {
        val events = mutableListOf<MyEvent>()
        val calendarEvents = queryCalendarEvents()
        
        for (calEvent in calendarEvents) {
            // 检查是否已存在（基于指纹去重）
            val fingerprint = EventFingerprint.generate(calEvent)
            if (!repository.isEventExists(fingerprint)) {
                events.add(CalendarEventMapper.toMyEvent(calEvent))
            }
        }
        
        return events
    }
}
```
